# RAG（检索增强生成）核心概念

## 什么是 RAG？

RAG = Retrieval-Augmented Generation，即检索增强生成。

**核心思想**：不让 LLM 仅靠自身知识回答问题，而是先从外部知识库检索相关文档，再将检索结果和问题一起交给 LLM 生成答案。

## 为什么需要 RAG？

| 问题 | RAG 解决方案 |
|------|-------------|
| LLM 知识过时 | 检索最新文档 |
| LLM 幻觉（胡说八道） | 基于真实检索内容回答 |
| LLM 缺乏专业知识 | 接入专业知识库 |
| 长文本处理成本高 | 只检索相关内容而非全部 |
| 无法访问私有数据 | 本地知识库检索 |

## RAG 核心流程

```
用户问题
   ↓
┌─────────────────┐
│  检索（Retrieve） │
└─────────────────┘
   ↓
┌─────────────────┐
│  重排序（Rerank）  │  ← 可选
└─────────────────┘
   ↓
┌─────────────────┐
│  生成（Generate） │
└─────────────────┘
   ↓
最终答案
```

## 关键组成部分

### 1. 知识源（Data Source）
- 文档：PDF、Word、Markdown、TXT
- 数据库：结构化/非结构化
- 网页抓取

### 2. 分块（Chunking）
将文档切成小块，是 RAG 效果的关键之一。

**常见策略**：
- 固定大小分块（如 500 token）
- 按段落/句子切分
- 递归字符分块（Recursive character split）
- 按语义分块（Small to large）

### 3. 向量化（Embedding）
将文本块转为向量，存入向量数据库。

### 4. 检索（Retrieval）
根据用户问题，从向量库中找出最相关的 Top-K 个块。

### 5. 生成（Generation）
将检索到的文档块 + 用户问题拼接成 prompt，送给 LLM 生成回答。

## 常见向量数据库

| 数据库 | 特点 |
|--------|------|
| Milvus | 开源、分布式、高性能 |

## 面试常问问题

**Q: RAG 和 Fine-tuning 的区别？**
- RAG：动态检索外部知识，更新知识无需重新训练
- Fine-tuning：用特定数据微调模型，修改模型权重

**Q: RAG 的局限性？**
- 检索质量依赖 Embedding 模型
- 上下文窗口限制（最多塞多少检索内容）
- 多跳推理能力弱（复杂问题效果差）

**Q: 如何提升 RAG 效果？**
- 优化分块策略（按语义而非固定大小）
- 用更好的 Embedding 模型（如 BGE、OpenAI embeddings）
- 增加重排序（Reranker）
- Query 改写/扩展（Query Rewriting）
- 混合检索（关键词 + 向量）
