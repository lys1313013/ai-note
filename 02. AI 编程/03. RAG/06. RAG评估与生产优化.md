# RAG 评估与生产优化

## RAG 评估体系

### 核心指标

| 指标 | 含义 | 评估方式 |
|------|------|----------|
| **Context Precision** | 检索结果中相关内容的排序质量 | Top-K 中相关项的排名 |
| **Context Recall** | 相关文档是否都被召回 | 与 Ground Truth 对比 |
| **Answer Faithfulness** | 回答是否基于检索内容（无幻觉） | 回答 vs 检索内容对齐 |
| **Answer Relevance** | 回答是否切中问题 | LLM 评分 / 用户打分 |
| **Hit Rate@K** | Top-K 中是否命中正确答案 | 精确匹配 |

### RAGAS 评估框架

RAGAS 是目前最流行的 RAG 评估库（https://github.com/explodinggradients/ragas）。

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness,      # 答案忠实度
    answer_relevancy,  # 答案相关性
    context_precision, # 上下文精确度
    context_recall     # 上下文召回率
)

results = evaluate(dataset, metrics=[faithfulness, answer_relevancy])
```

评估维度：
- **Faithfulness**：生成回答中的每个陈述是否都能从检索块中找到依据
- **Answer Relevancy**：回答是否直接回应问题
- **Context Precision**：Top-K 排序质量
- **Context Recall**：是否漏掉了重要信息

## 生产环境关键问题

### 1. 延迟（Latency）

| 优化方向 | 方法 |
|----------|------|
| Embedding 加速 | 模型量化（INT8/INT4）、batch 推理 |
| 检索加速 | HNSW 索引、量化向量 |
| 生成加速 | 小模型 + Short context |

### 2. 成本控制

- Embedding 模型选型：本地 BGE vs OpenAI（cost vs quality）
- 上下文压缩减少 token
- 缓存已检索的 Query 结果

### 3. 多语言/混合内容

- 文档含中英文混杂 → 需支持多语言 Embedding（BGE 支持）
- 表格、图表 → 需专用解析（table-caption 对齐）

### 4. 实时知识更新

| 方案 | 适用场景 |
|------|----------|
| 定时增量索引 | 知识更新不频繁 |
| 流式更新 + 版本管理 | 实时性要求高 |
| 长窗口 LLM 直接读文档 | 更新极频繁（如股票） |

## 常见失败模式

| 模式 | 原因 | 解决方案 |
|------|------|----------|
| 检索不到 | 分块太大/Embedding 不合适 | 优化 Chunking |
| 幻觉严重 | 检索内容不够，LLM 胡编 | Prompt 约束 + Citations |
| 回答片面 | 只召回部分相关块 | 提升 Recall（混合检索） |
| 重复内容 | 多个块相似度高 | 去重/Deduplication |
| 速度慢 | 向量库/HNSW 参数不对 | 调优索引参数 |

## 面试常问问题

**Q: 如何评估 RAG 系统效果？**
RAGAS 四件套（Faithfulness、Relevancy、Precision、Recall），加端到端准确率。

**Q: 生产环境 RAG 的最大挑战？**
检索质量和生成质量的平衡，以及延迟和成本的权衡。

**Q: 如何处理实时性要求高的知识？**
定期重建索引 + 长上下文 LLM 直接读文档做兜底。
