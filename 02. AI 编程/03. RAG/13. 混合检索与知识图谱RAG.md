# 混合检索与知识图谱 RAG

## 混合检索（Hybrid Search）

### 为什么需要混合？

| 检索方式 | 优势 | 劣势 |
|----------|------|------|
| 向量检索 | 语义理解强、同义词好 | 精确关键词匹配弱 |
| 关键词检索（BM25） | 精确匹配、专有名词好 | 语义泛化能力差 |

两者结合，取长补短。

### 实现方式：RRF（Reciprocal Rank Fusion）

```python
def hybrid_search(query, vector_results, bm25_results, k=60):
    """
    RRF 融合公式：
    score(d) = Σ 1/(k + rank(d))
    """
    scores = defaultdict(float)
    
    for rank, doc in enumerate(vector_results):
        scores[doc.id] += 1 / (k + rank)
    
    for rank, doc in enumerate(bm25_results):
        scores[doc.id] += 1 / (k + rank)
    
    # 按融合分数排序
    sorted_docs = sorted(scores.items(), key=lambda x: -x[1])
    return sorted_docs
```

### Alpha 加权融合

```python
def weighted_hybrid(alpha, vector_scores, bm25_scores):
    return alpha * vector_scores + (1 - alpha) * bm25_scores
    # alpha = 0.7 表示向量检索权重 70%
```

## 知识图谱 RAG（Graph RAG）

### 核心思想

传统 RAG 检索的是**文本块**，Graph RAG 检索的是**实体和关系构成的知识图谱**。

```python
# 知识图谱示例
(Godfather {name: "教父"}) ←[:HAS_CHARACTER]─ (Movie {title: "教父"})
(Al Pacino {name: "阿尔·帕西诺"}) ←[:ACTED_IN]─ (Movie {title: "教父"})
```

用户问"教父的主演是谁？" → 检索图谱中教父和演员的关系 → 生成答案。

### 构建知识图谱

```python
# 用 LLM 提取实体和关系
def extract_graph(text):
    prompt = """
    从以下文本中提取实体和关系，以三元组形式返回：
    实体1, 关系, 实体2
```

### Graph RAG vs Vanilla RAG

| 维度 | Vanilla RAG | Graph RAG |
|------|-------------|-----------|
| 检索单元 | 文本块 | 实体+关系 |
| 推理能力 | 弱（依赖文本顺序） | 强（关系网络推理） |
| 多跳问答 | 差 | 好 |
| 构建成本 | 低 | 高（需要 LLM 抽取图谱） |
| 适用场景 | 简单问答 | 复杂关系推理 |

### 微软 GraphRAG

微软 2024 年提出的方案，核心思想：
1. 构建实体知识图谱（Entity Graph）
2. 按实体聚类分组（Communities）
3. 生成每个 Community 的摘要（Community Summary）
4. 检索时先查 Community Summary，再深入细节

优势：**大幅减少 token 消耗**，适合大规模文档理解。

## 面试常问问题

**Q: 混合检索的融合算法有哪些？**
RRF（倒数排序融合）和 alpha 加权最常见。RRF 优点是不需要训练，直接根据排名融合。

**Q: Graph RAG 什么时候用？**
需要多跳推理（查关系）、文档之间有明确实体关系（人物/公司/产品）、需要理解事物之间的逻辑链时。

**Q: Graph RAG 的构建成本怎么优化？**
用 LLM 抽取时控制 prompt 长度+批次处理；实体去重（用 Embedding 相似度合并）；优先构建核心实体的图谱。
