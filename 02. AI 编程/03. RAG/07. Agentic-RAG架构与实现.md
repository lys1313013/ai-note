# Agentic RAG 架构与实现

## 核心理念

Agentic RAG 的本质是：**把 LLM 变成 RAG 流水线的编排者（Orchestrator）**，让它决定：
- 何时检索
- 检索什么
- 如何利用检索结果
- 是否需要补充检索

## 架构对比

```
传统 RAG（Pipeline 模式）：
User Query → [Retrieve] → [Generate] → Answer
              固定流程，顺序执行

Agentic RAG（Agent 模式）：
User Query → [Agent] ← tools(RAG, Web, DB)
                  ↓
              decide → action → observe → decide...
```

Agent 是一个循环：决策→行动→观察→再决策。

## 关键组件

### 1. Agent（决策者）
- 理解用户意图
- 规划检索策略
- 评估结果质量
- 决定是否需要更多检索

### 2. Tools（工具集）
```python
tools = [
    "vector_search",      # 向量库检索
    "keyword_search",      # BM25/Elasticsearch
    "web_search",         # 搜索引擎兜底
    "sql_query",          # 结构化数据库
    "rewrite_query",      # Query 改写
    "calculate",          # 计算器
]
```

### 3. Memory（记忆）
- 短期：当前对话的检索历史
- 长期：跨会话知识积累

### 4. State（状态机）
跟踪每轮检索的结果，决定下一步。

## 状态机设计

```python
class RAGState(TypedDict):
    query: str                      # 当前问题
    retrieved_docs: List[Document]  # 已检索的文档
    generation: str                  # 当前/上次生成内容
    iterations: int                 # 迭代次数
    need_more_info: bool            # 是否需要补充检索
    answer: str                     # 最终答案
```

典型流程：
```
Start → Plan → Retrieve → Generate → Eval → 
  ↑                         ↓
  └──── 不足则 loop ────────┘
```

## 面试常问问题

**Q: Agentic RAG 和普通 RAG 的本质区别？**
普通 RAG 是一个固定流程（retrieve→generate），Agentic RAG 是一个循环决策过程，Agent 能根据中间结果动态调整策略。

**Q: Agentic RAG 的典型应用场景？**
- 复杂多跳问答（需要多步推理）
- 需要外部工具配合（搜索+数据库+RAG）
- 开放式问答（不知道需要多少步检索）
- 需要自我纠错的场景
