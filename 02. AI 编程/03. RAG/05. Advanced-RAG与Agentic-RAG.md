# Advanced RAG 与 Agentic RAG

## 基础 RAG → Advanced RAG

```
基础 RAG: Query → Retrieve → Generate
         ↓
Advanced: Query → Pre-Retrieval → Retrieve → Post-Retrieval → Generate
                              ↓                ↓
                         Query改写           Rerank/压缩
```

## Agentic RAG（智能体 RAG）

核心思想：RAG 系统不再只是"检索+生成"的流水线，而是变成一个**有决策能力的 Agent**。

### Agentic RAG 的能力

- **判断是否需要检索**：简单问题可直接回答
- **决定检索什么**：多跳问题分解为多个子查询
- **评估检索结果**：检查召回内容是否足够，不够则补充检索
- **自我纠错**：生成答案后检查是否 hallucination

### 典型框架

```
Agent
  ├── 判断：需要检索？ → 规划子查询
  ├── 工具：RAG检索、搜索引擎、数据库查询
  ├── 评估：结果是否足够？
  │     ↓ 不够 → 补充检索
  └── 生成：综合所有检索结果回答
```

### 代表性工作

| 论文/项目 | 核心思想 |
|-----------|----------|
| Self-RAG（斯坦福） | 模型自检索、自评估、自纠正 |
| REFEED | 生成后检查幻觉，补充检索 |
| Corrective-RAG | 评估检索结果质量，失败则用搜索引擎兜底 |
| Agentic-RAG（LangGraph） | 用状态机控制多跳检索流程 |

## Agentic RAG vs 普通 RAG

| 维度 | 普通 RAG | Agentic RAG |
|------|---------|-------------|
| 检索次数 | 固定1次 | 按需多次 |
| 查询处理 | 一次性 | 分解+改写 |
| 结果评估 | 无 | 有，失败重试 |
| 外部工具 | 仅向量库 | 搜索+数据库+计算器 |
| 适用场景 | 简单单跳 | 复杂多跳 |

## 面试常问问题

**Q: 什么是 Agentic RAG？**
把 RAG 的检索和生成流程做成一个 Agent 的决策过程，能根据中间结果动态决定下一步（检索什么、是否需要补充、是否需要换查询策略）。

**Q: Self-RAG 和普通 RAG 的区别？**
普通 RAG 只管检索，Self-RAG 让模型自己判断：检索是否必要、检索结果是否有用、回答是否幻觉。

**Q: RAG 系统如何应对幻觉？**
- 强制 LLM 基于检索结果回答（prompt 约束）
- 引用溯源（citations），让用户可核对
- Self-RAG 自检机制
- 后置幻觉检测（如NnNC）
