# RAG 分块策略（Chunking）

## 为什么分块很重要？

分块决定了检索的最小单元。分块太大 → 包含太多无关信息；分块太小 → 丢失上下文。

## 分块策略

### 1. 固定大小分块（Fixed Size）
```python
# 每块 500 token，重叠 50 token
chunk_size = 500
chunk_overlap = 50
```
- 简单粗暴，适用面广
- 缺点：可能切断语义连贯性

### 2. 递归字符分块（Recursive Character Split）
按段落 → 句子 → 单词逐级拆分，保持语义完整性。

### 3. 按语义分块（Semantic Splitting）
用 LLM 或 Embedding 判断句子之间的语义相似度，在断点处切分。

### 4. 小到大策略（Small to Large）
```python
small_chunk = 200 token   # 检索用（精准）
large_chunk = 1000 token  # 生成用（完整上下文）
```
检索用小块保持精准，生成时扩展到更大窗口获取上下文。

## 分块参数选择

| 参数 | 影响 |
|------|------|
| chunk_size | 太小丢失上下文，太大引入噪声 |
| overlap | 边界信息丢失，需要重叠保证连续性 |

**一般经验**：
- 通用文档：500-1000 token/块
- 代码：函数级别（更小块）
- 论文/报告：段落级别

## 分块质量判断

好的分块：
- ✅ 每个块有独立完整的语义
- ✅ 块不要包含太多不同主题
- ✅ 保留块之间的上下文关联

## 面试常问问题

**Q: 如何选择分块大小？**
没有标准答案，需要实验。通常 500-1000 token 是起点，根据具体文档类型和任务调整。

**Q: 块重叠（overlap）的作用？**
防止边界信息丢失。相邻块之间保持 10-20% 重叠。
